# Purifier — TUI Disk Cleanup Tool

A terminal-based disk space analyzer and cleanup tool for macOS, inspired by GrandPerspective. Built with Rust + Ratatui. Features a hybrid safety classifier (built-in rules + LLM via OpenRouter) that explains whether each path is safe to delete.

## Decisions

| Decision | Choice |
|----------|--------|
| Audience | Both developers + general users |
| Stack | Rust + Ratatui |
| Architecture | Layered: `purifier-core` library + `purifier-tui` binary |
| Safe-to-delete | Hybrid: built-in rules + OpenRouter LLM for unknowns |
| Scan scope | Full disk default, optional path targeting |
| Deletion model | Inline per-item confirm + smart "suggested cleanup" by category |
| Views | Tabbed: by size, by type, by safety, by age |
| Platform | macOS only (day one) |

## Data Model

```rust
struct FileEntry {
    path: PathBuf,
    size: u64,
    is_dir: bool,
    modified: SystemTime,
    category: Category,        // Cache, BuildArtifact, Download, AppData, Media, System, Unknown
    safety: SafetyLevel,       // Safe, Caution, Unsafe, Unknown
    safety_reason: String,     // "node_modules — regenerated by `npm install`"
    children: Vec<FileEntry>,
}
```

### Categories

- **BuildArtifact** — `node_modules/`, `target/`, `.build/`, `__pycache__/`, etc.
- **Cache** — `~/Library/Caches/*`, `~/.cache/`, browser caches
- **Download** — `~/Downloads/`, `.dmg`, `.pkg`, `.zip` installers
- **AppData** — `~/Library/Application Support/*`, app databases, settings
- **Media** — large video/audio/image files
- **System** — OS files, `.git/objects`, system libraries
- **Unknown** — anything not matched by rules

### Safety Levels

- **Safe** (green) — regenerable, cached, or already trashed
- **Caution** (yellow) — likely safe but verify (e.g., old downloads, large media)
- **Unsafe** (red) — deletion causes data loss (app data, git history, documents)
- **Unknown** (gray) — no rule matched, awaiting or failed LLM classification

## Scanner

- Uses `jwalk` for parallel filesystem walking
- Results stream to TUI via `crossbeam-channel` as they arrive — UI updates progressively
- Each entry classified by rules engine immediately on arrival
- Directory sizes computed bottom-up as child entries arrive
- Entries with `safety: Unknown` batched (up to 50 per request) and sent to LLM asynchronously
- UI shows "analyzing..." for pending LLM entries, updates in place when verdict arrives

## Safety Rules Engine

Rules defined in `rules/default.toml`:

```toml
[[rules]]
pattern = "node_modules"
category = "BuildArtifact"
safety = "Safe"
reason = "Regenerated by `npm install` from package.json"

[[rules]]
pattern = "target/debug"
category = "BuildArtifact"
safety = "Safe"
reason = "Rust debug build output — regenerated by `cargo build`"

[[rules]]
pattern = "~/Library/Caches/*"
category = "Cache"
safety = "Safe"
reason = "Application cache — apps recreate as needed"

[[rules]]
pattern = "~/.Trash/*"
category = "Cache"
safety = "Safe"
reason = "Already in trash — safe to permanently delete"

[[rules]]
pattern = "~/Downloads/*.dmg"
category = "Download"
safety = "Caution"
reason = "Disk image — likely safe if app is already installed"

[[rules]]
pattern = "~/Library/Application Support/*"
category = "AppData"
safety = "Unsafe"
reason = "App data and settings — deletion may cause data loss"

[[rules]]
pattern = ".git/objects"
category = "System"
safety = "Unsafe"
reason = "Git object store — deletion destroys repository history"
```

**Matching:** top-to-bottom, first match wins. Patterns support globs (`*`, `**`) and `~` expansion. Users can supply additional rules via `--rules extra.toml` (evaluated before defaults).

## LLM Classifier (OpenRouter)

- Batches up to 50 unknown paths per request
- Sends: path, parent context, file size, age
- Asks for: category, safety level, one-line explanation
- Results cached in memory for the session (same directory patterns don't re-query)
- If OpenRouter unreachable: entry stays `Unknown` with reason "Could not classify — review manually"
- API key via `--api-key` flag or `OPENROUTER_API_KEY` environment variable
- Can be disabled entirely with `--no-llm`

Four tabbed views, switchable via `1`/`2`/`3`/`4` keys:

```
│──────────────────────────────────────────────────────────────────│

│ [Enter] expand  [d] delete  [?] explain  [/] search  [q] quit   │
└──────────────────────────────────────────────────────────────────┘
```

- Tree navigation: arrow keys + vim keys (`j`/`k`/`h`/`l`)
- Enter to expand/collapse directories
- Size bars proportional to parent
- Colored inline safety badge (green ✓, yellow ⚠, red ✗, gray ?)
- Bottom panel: safety explanation for highlighted item

### Tab 2: By Type

Collapsible sections: Build Artifacts, Caches, Downloads, App Data, Media, System, Unknown — each showing total size and item count.

### Tab 3: By Safety

Three sections: Safe to Delete (green), Use Caution (yellow), Do Not Delete (red), plus Unknown/Analyzing. This is the "suggested cleanup" view — user can approve/reject entire groups or individual items.

### Tab 4: By Age

Sorted by last modified. Highlights files untouched for 6mo+, 1yr+, 2yr+.

## Deletion Flow

1. Press `d` on any item (in any view)
2. Confirmation popup: path, size, safety level, reason, `[y/n]`
3. On confirm: item deleted, tree updates, freed space counter increments in status bar

## CLI Interface

```
purifier                     # scan full disk
purifier ~/Projects          # scan specific path
purifier --rules my.toml     # add custom rules
purifier --no-llm            # skip LLM classification
purifier --api-key KEY       # OpenRouter key (also reads OPENROUTER_API_KEY env)
```

## Error Handling

- **Permission denied during scan:** skip entry, log, show counter ("12 paths skipped — no permission") in status bar
- **OpenRouter unreachable:** entries stay `Unknown`, status bar shows "LLM offline — some paths unclassified". Tool fully usable without LLM.
- **Deletion fails:** inline error ("Could not delete: permission denied"), item stays in tree
- **Disk full / IO errors:** graceful message, scan continues for accessible paths

## Testing Strategy

- **Unit tests** in `purifier-core`: rules matching, size aggregation, category assignment, LLM response parsing
- **Integration tests**: mock filesystem (tempdir) + mock OpenRouter (wiremock) to test full scan→classify→display pipeline
- **Manual testing**: run against `~/` on macOS, verify safety classifications make sense, test deletion on safe targets
